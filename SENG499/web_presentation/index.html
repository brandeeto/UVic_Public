The Event

During the week of March 14th, The UVic Engineering Student Society (ESS) runs The Order of Pi, an event where students, staff, and alumni can be pied by their peers by donation. All proceeds of the event are donated to the Children’s Health Foundation of Vancouver Island.

Being a student run event, the organizers oftentimes do not the time needed to run the order of pi event. Currently donations are in-person cash payments, volunteers are scheduled by pen and paper and the system organizing the event contracts is on a archaic system that is not maintained.

To give back to the university and the Engineering Student Society, this project focuses on the event organizational inefficiencies and finds the appropriate optimizations and suistainable technologies at a student budget.

The Project

The project, Re-Order of Pi, had the main goal of optimizing the background processes for ESS event organizers while creating an interally maintainable application for future considerations. The key components to this include:



Contract Management


Donation Process

We are a team building a product for the ESS to improve their ability to run the Order of Pi, a week long event that involves many logisitical elements. Understanding the workflow of the event is key to building the right product for the event. Jason Syrotuck, our team lead, has acted as Charity Coordinator for the ESS on two occasions, and was responsible for the events operations. We consulted with other past Order of Pi organizers and developed a functional requirements matrix to help describe what the web app needs to do, and prioritize the different aspects, this was the first major step in our design phase.

After understanding what the web application needs to do, we developed an entity relationship diagram to describe the database tables and columns. In conjunction with the ER diagram we developed a simple workflow diagram, showing the connections and interactions between pages.

The development language was key for maintainability, we considered Python first, favoring readability over all other considerations. Django, a Python web application framework, some group members had experience with Django, and all were comfortable with Python, making it the ideal choice for development.

Technologies


To create a base layer that other components can be built upon, many web applications use a framework for rapid development while using standardized base code. Although many frameworks exist, Django was an obvious choice due to our team’s background with Python. It offered an extendable solution while offering protection against many security flaws that exist in web applications such as Cross Site Scripting, Cross Site Forgery Requests, and SQL Injection to name a few. Furthermore, Django offers many different versions that have been evolving over the years. The Order of Pi web application needs to have maintained solutions so that future students involved with the Engineering Student Society are not constantly concerned with security and data-loss patches. This led to the decision to use Django version 1.11LTS, which is the latest Long Term Support solution offered by Django. This version will be maintained until April 2020 with feature releases every 8 months as well as applied security and data-loss fixes.


The main reasons for rebuilding the Order’s site was to make it easier for the organizers to run, and to bring in online payments to make it easier and more convenient for people to donate. Many options were researched with regards to online payments but Stripe was chose due to the ease of integration as well as having the lowest fees. The stripe checkout widget is used to verify and validate the credit card information entered, this also keeps all of the sensitive credit card information off the site.


The webserver for the Order's site consists of two separate modules, working hand-in-hand. The first of which is the uWSGI module; tasked with packaging and serving the Django Python application in a format that front-end webservers can understand. NGINX is the front-facing component of the webserver; efficiently handling the user's requests, providing load balancing, and leveraging the standard in encryption technology to ensure all transactions are secured. uWSGI and NGINX communicate directly through their own server-side port, together forming the Order's webserver.


For this project the team was hoping to use Postgres as a query language as it is what we had learned in school, but it was far easier to work with the existing setup. The ESS runs a MySQL database on thier server which is version 5.5 so everything implemented needed to be able to run on that. A schema with five new tables was created to facilitate the development of the project.


Cloudflare provides the Order's SSL certificates, giving us the ability to properly authenticate with our users. Having properly signed SSL certificates is a fundamental foundation for any trusted website leverage the HTTPS protocol. This gives the users peace-of-mind knowing that all transactions, including credit card transactions, remain secure.

There are some critical steps that must be taken when replacing an existing system and workflow with a new one. First we must examine the existing system, the functionality that exists, and how it is currently used. Secondly is understanding the motivation for the new system, what functionality is missing? What existing workflows are not effective? After that we design and outline the new system, what it will do, and what other systems it will integrate with, and bringing that design to the product owners to ensure we are building the right product. Lastly is to design and implement the components of the system, starting with the most core components and the less critical aspects.


  • Project Milestones

  • Milestone 0 - Role Assignment
    • Internal roles assigned to team members based on experience.
  • Milestone 1.1 - Requirements Gathering
    • Requirements elicication to find the project scope. Requirements were then documented and prioritized.
  • Milestone 1.2 - Tool Research
    • Tools and technologies researched that will allow the application to work with current the current ESS system as well as be maintainable in the future.
  • Milestone 1.3 - Development Environment Set-Up
    • Tools and technologies researched that will allow the application to work with current the current ESS system as well as be maintainable in the future.
  • Milestone 2 - Documentation and Requirements Finalization
    • The documentation surrounding the functional requirements of the website, the functionality of the database, and the outline of the flow for the website is to be completed. This will allow the development of the site to proceed quickly due to the major issues being worked out first.
  • Milestone 3 - Architecture Skeleton and Dependencies
    • A broad start towards the minimum viable product, the skeleton work including all views, url tree, HTML templates, and page dependencies completed across all required pages of the website.
  • Team review of development environments and team architecture review.
  • Milestone 4 - Minimum Viable Product (MVP)
    • Critical components of the website functioning such as user log-in and contract management actions. Online payments were not mandatory for this milestone, but were reviewed.
  • Milestone 4 - Minimum Viable Product (MVP)
    • Critical components of the website functioning such as user log-in and contract management actions. Online payments were not mandatory for this milestone, but were reviewed.
  • Milestone 5 - Medium Priority Requirements and Team Webpage
    • Implement requirements that are below critical, such as requirements with priority 2 or 3. These requirements included the payment processing, volunteer signup and email templates.
  • Team review of application to this point to ensure meeting requirements.
  • Milestone 6 - Low Priority Requirements and Finalizations
    • A buffer for outstanding items that occur from previous milestones as well as an opportunity to complete low priority requirements. This milestone also included having the SSL certifications, NGNIX and uWSGI communicating on the server for the public presentations.
